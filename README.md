# Reel

A small pseudo-language (DSL) for describing **Reels / Shorts / TikTok**
short-form-video scenarios, with a parser generated by **ANTLR 4**.

A `.reel` file describes the *structure* of a vertical video ŌĆö its metadata,
an ordered list of timed scenes, and per-scene properties for visuals,
narration, on-screen text, audio, transitions and effects. It is designed to
be writable by humans (creators, prompt engineers, story-boarders) and
machine-parseable by downstream tooling (auto-editors, schedulers, voiceover
generators, storyboarding renderers).

```reel
reel "Morning Routine Hacks" {
  meta {
    platform: instagram
    aspect:   9:16
    duration: 30s
    music:    "Sunny Lo-Fi" by "DJ Apricot"
    tags:     #fyp #morningroutine
  }
  scene "hook" (0s - 3s) {
    text:       "POV: it's 6am" #big #center
    visual:     "person slams alarm"
    transition: cut
  }
}
```

## Quick start

Requires Java 17+ and `antlr4` (Homebrew: `brew install antlr`).

```bash
make              # generate + compile the parser and the JSON listener
make test         # parse every examples/*.reel, fail on any error
make test-errors  # parse every examples/broken/*.reel, fail if none errors
make json F=cookbook    # emit a JSON timeline (see "Consuming")
make tree F=cookbook    # print the parse tree for examples/cookbook.reel
make tokens F=hello     # dump the token stream
make gui F=cookbook     # graphical parse-tree viewer
make clean        # remove generated artifacts
```

Files:

| File              | Purpose                                                  |
|-------------------|----------------------------------------------------------|
| `Reel.g4`         | The ANTLR4 grammar (lexer + parser).                     |
| `ReelToJson.java` | Reference consumer ŌĆö walks the tree, emits JSON.         |
| `examples/`       | Valid programs: `hello.reel`, `cookbook.reel`.           |
| `examples/broken/`| Intentionally invalid programs (negative tests).         |
| `Makefile`        | Generate / compile / test harness.                       |

## Language reference

### Program structure

A program is a sequence of **statements**. The common case is one or more
video declarations:

```
<videoKind> "<title>" { <statement>* }
```

`<videoKind>` is one of `reel`, `short`, `video`, `tiktok`. The title is
optional (`reel { ... }` is valid). The body is a block of statements.

### Statements

| Statement   | Syntax                                            |
|-------------|---------------------------------------------------|
| Video       | `reel "Title" { ... }`                            |
| Scene       | `scene "name" (timing) { ... }`                   |
| Meta block  | `meta { ... }` (convention: platform/aspect/...)  |
| Property    | `key: value`                                      |
| Constant    | `define NAME = value`                             |
| Import      | `import "shared.reel"` (path only ŌĆö see Imports)  |

### Scenes and timing

A scene's name and timing are both optional:

```reel
scene              { ... }   // anonymous, ordered
scene "hook"       { ... }   // named
scene (3s)         { ... }   // duration only
scene (0s - 3s)    { ... }   // explicit start..end (dash form)
scene (3s .. 5s)   { ... }   // explicit start..end (dot form)
```

Time points are **durations** with a unit: `30s`, `500ms`, `2m`, `1.5s`, `1h`.

> **Units are required in timing.** A bare number is rejected in a scene
> window ŌĆö `scene (0 - 3s)` is a parse error, because "0" is ambiguous
> (seconds? frames?). Write `scene (0s - 3s)`. Bare numbers are still fine
> as ordinary values (`fps: 30`, `speed: 1.5`).

### Values

A property value can be any of:

| Kind        | Example                          | Notes                                   |
|-------------|----------------------------------|-----------------------------------------|
| String      | `"hi"`, `'hi'`, `"caf├® ŌśĢ"`       | Single line; escapes `\n \t \" \\`.     |
| Duration    | `30s`, `500ms`, `2m`, `1h`       | Number + unit.                          |
| Number      | `30`, `1.5`                      | Integer or decimal.                     |
| Aspect      | `9:16`, `1:1`                    | First-class value (one token).          |
| Boolean     | `true`, `false`                  |                                         |
| Identifier  | `cut`, `fade_out`, `zoom-in`, `├╝ber` | Open vocabulary; **Unicode-aware**. |
| Reference   | `$MAX_LEN`                       | Resolves to a `define`.                 |
| List        | `["a", "b", "c"]`                | Trailing comma allowed.                 |
| Tags        | `#fyp`, `#2025goals`, `#ą┐čĆąĖą▓č¢čé`  | May start with a digit; **Unicode-aware**. |

Identifiers and tags are Unicode: `caption: caf├®`, `#100daychallenge`,
`#ą┐čĆąĖą▓č¢čé` all lex correctly. (Emoji and other symbols are not identifiers ŌĆö
put them inside a string.) Strings are **single-line**: a backslash cannot
continue a string across a newline, so an unterminated quote is reported on
the line where it occurs rather than running away to EOF.

**Attribution** ŌĆö `by` joins a media value to its author (string or `$ref`):

```reel
music: "Sunny Lo-Fi" by "DJ Apricot"
music: "Beat"        by $ARTIST
```

**Style tags** ŌĆö any value may carry trailing `#tags`:

```reel
text: "POV: it's 6am" #big #center
```

**Multiple values** ŌĆö comma-separated after one key:

```reel
sfx: "whoosh", "boop", "pop"
```

### Constants

```reel
define MAX_LEN  = 60s
define ARTIST   = "DJ Apricot"

reel "x" {
  duration: $MAX_LEN
  music:    "song" by $ARTIST
}
```

`define`s may appear anywhere (including inside a scene), but their
**scope is the whole file**: all defines are collected into one global table,
**the last definition of a name wins**, and references are resolved *after*
the full file is parsed, so **forward references work** (`$X` may appear
before `define X`). The reference listener (`ReelToJson`) implements exactly
this; see "Semantics" below.

### Imports

```reel
import "shared.reel"
```

`import` records a path to another `.reel` file (typically one holding shared
`define`s or a `meta` block). The grammar **only parses the path** ŌĆö loading
the file, merging its contents and handling cycles are the consumer's job.
The reference listener ignores imports.

## Semantics the grammar leaves open

The grammar is deliberately structural; a few things are not (and cannot be)
expressed in it. These rules are pinned down here so every consumer agrees,
and the reference listener implements them:

| Concern            | Rule                                                                 |
|--------------------|----------------------------------------------------------------------|
| `define` scope     | File-scoped global table; **last write wins**; forward refs allowed. |
| Duplicate keys     | **Accumulate** in source order into an ordered list (layered text).  |
| Scene order        | Source order. There is no `order:` field; reorder by editing the file.|
| Timing units       | Required (`0s`, not `0`); a `$ref` must resolve to a duration.       |
| Import resolution  | Consumer-side; the grammar records the path only.                    |
| Unknown property   | Allowed and passed through ŌĆö the vocabulary is open.                 |

### Comments

```reel
//  line comment
/*  block comment (does NOT nest) */
```

> `#` is **not** a comment marker ŌĆö it begins a style tag (`#word`). Use `//`.

### Reserved words

`reel`, `short`, `video`, `tiktok`, `scene`, `meta`, `by`, `define`, `import`,
`true`, `false`. They cannot be used as property *keys*, but **can** be used as
plain *values* (e.g. `format: short`).

## Design notes

* **Open vocabulary.** Transitions (`cut`, `whip_pan`), effects (`zoom-in`,
  `slow-mo`), platforms (`instagram`) and arbitrary properties are just
  identifiers ŌĆö new terms need no grammar change.
* **Aspect as one token.** `9:16` is lexed as a single `ASPECT` token, so its
  inner `:` never collides with the key/value separator. (Only integer ratios;
  named presets like `vertical` are just identifiers, and fractional ratios
  like `9:16.5` are not supported ŌĆö by design.)
* **No arithmetic.** Deliberately omitted; durations and times are literals or
  `$refs`, keeping the grammar unambiguous and creator-friendly.
* **Permissive top level.** Bare scenes, properties, `define`s and `meta`
  blocks parse anywhere, so partial files and config snippets are legal.

## Consuming the parse tree

`ReelToJson.java` is a working consumer: it extends `ReelBaseListener`, walks
the tree, resolves `$refs`, and prints a JSON timeline. Run it:

```bash
make json F=cookbook        # or: java -cp "$ANTLR_JAR:." ReelToJson examples/cookbook.reel
```

```json
[{ "kind": "reel", "title": "Morning Routine Hacks",
   "meta": { "platform": "instagram", "aspect": "9:16", "duration": "60s",
             "music": { "value": "Sunny Lo-Fi", "by": "DJ Apricot" },
             "tags": ["fyp","morningroutine","viral"], "autoplay": true },
   "scenes": [
     { "name": "hook", "timing": { "start": "0s", "end": "3s" },
       "props": { "text": { "value": "POV: it's 6am", "tags": ["big","center"] },
                  "sfx": ["whoosh","boop"], "transition": "cut" } },
     { "name": "beat", "timing": { "duration": "500ms" }, ... }
   ] }]
```

Note how it demonstrates the open semantics: `$MAX_LEN`ŌåÆ`60s`, the `by`
clause becomes `{value, by}`, duplicate `sfx` values become an array, and a
single value collapses to a scalar while `scenes` stays a list. Use it as a
starting point for your own renderer/exporter.
